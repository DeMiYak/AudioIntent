# Отчёт: извлечение выражений установления и прекращения контакта

## 1. Постановка задачи

Задача проекта — автоматически найти в русскоязычном фильме реплики, которые устанавливают или завершают контакт между персонажами, и представить результат в виде таблицы `Спикер - фраза` с разделением на `opening` и `closing`.

К таким репликам относятся не только прямые приветствия и прощания, но и прагматические реплики начала или завершения взаимодействия, если они размечены в gold как `contact_open` или `contact_close`.

Вход: видео- или аудиофайл фильма без разметки. Выход: Excel-файл `extracted_pairs.xlsx` с колонками `opening` и `closing`. Для просмотра результата по событиям также используется `detailed_pairs.xlsx`.

---

## 2. Реализованное решение

### 2.1. Подготовка обучающих данных

Из листа `Диалоги` файла `data/raw/gold/data_val.xlsx` подготовлен JSONL-файл:

```text
data/processed/gold_dialogues.jsonl
```

В нём 1170 реплик из 18 фильмов:

| Класс | Количество |
|---|---:|
| `contact_open` | 413 |
| `contact_close` | 175 |
| `none` | 582 |

Подготовку выполняет модуль `src/preprocess_gold.py`. Он извлекает теги `<opening>` и `<closing>` из аннотированного текста, сохраняет нормализованные записи и статистику.

### 2.2. Rule-based классификатор

Rule-based часть реализована в `src/rule_based_intent.py`.

Она состоит из трёх элементов:

1. Лексикон фраз, автоматически извлечённый из gold-разметки.
2. Сильные и слабые маркеры opening/closing.
3. Ручные регулярные выражения `MANUAL_PATTERNS`.

Лексикон строится командой `extract_lexicon_from_gold()` и компилируется функцией `compile_lexicon()`. Для поиска кандидатов используются функции `collect_candidates_for_text()`, `score_candidate()` и `expand_match_to_phrase()`.

Ручные правила покрывают как прямые формулы контакта (`здравствуйте`, `до свидания`, `увидимся`, `чао`), так и отдельные прагматические выражения, важные для текущей разметки (`желаете что-нибудь`, `я пошёл`, `буду через полчаса`, `мы снимаемся с соревнований`).

В текущем коде все ручные правила из `OPEN_MANUAL_RULES` и `CLOSE_MANUAL_RULES` считаются специальными: в combined-режиме они не требуют подтверждения ML-моделью.

### 2.3. ML-классификатор

ML-компонент реализован в `src/ml_intent.py` и обучается через `src/train_intent_model.py`.

Используется схема:

```text
TF-IDF по символьным n-граммам 2–5 + sklearn-классификатор
```

Поддерживаемые модели:

```text
LogisticRegression, LinearSVC + calibration, ComplementNB, SGDClassifier, RidgeClassifier + calibration
```

Для рабочей конфигурации используется RidgeClassifier с `CalibratedClassifierCV`, обученный с параметром:

```bash
--classifier ridge
```

ML-признаки:

- символьные n-граммы текста;
- относительная позиция реплики в диалоге;
- длина реплики;
- флаги начала и конца диалога;
- наличие маркеров opening/closing в предыдущей и следующей реплике.

Контекстные признаки добавлены потому, что контактные реплики часто образуют локальные пары: приветствие может сопровождаться ответным приветствием, а прощание — ответным завершением контакта.

### 2.4. Combined-режим

Combined-режим реализован в функции `_run_intent_extraction()` файла `src/pipeline.py`.

Точная логика:

1. Сначала независимо запускаются rule-based и ML-компоненты.
2. ML-предсказания ниже `--ml-confidence-threshold` удаляются.
3. Rule-based предсказание сохраняется, если ML дал тот же тип для той же реплики.
4. Исключение: ручные правила из `OPEN_MANUAL_RULES` и `CLOSE_MANUAL_RULES` сохраняются без ML-подтверждения.
5. ML-предсказания добавляются к результату, если такая пара `(utterance_id, intent_type)` ещё не была добавлена.
6. Если одна реплика получила обе метки (`contact_open` и `contact_close`), rule-based метка удаляется только тогда, когда ML дал противоположный тип.

Таким образом, combined-режим — это не простая сумма двух систем. Это объединение с ML-фильтром для большинства rule-based результатов и исключением для ручных правил.

### 2.5. ASR, diarization и сборка реплик

ASR выполняется через faster-whisper, diarization — через pyannote. Эти этапы вынесены в Colab-ноутбуки:

```text
notebooks/google_colab_asr_pipeline.ipynb
notebooks/google_colab_diarization_pipeline_venv.ipynb
```

Результаты сохраняются в папках `windows/`, где каждое окно содержит:

```text
transcript.json
diarization.json
audio.wav
```

Сборка реплик выполняется в `src/utterance_builder.py`. Для сопоставления ASR и diarization используются временные границы слов или сегментов. В рабочей validation-команде используется:

```bash
--diarization-segment-mode regular
```

Этот режим берёт `regular_segments` из `diarization.json`, если они доступны.

### 2.6. Speaker identification

Speaker identification реализован в `src/speaker_id.py`.

Используется Resemblyzer `VoiceEncoder`. Для каждого персонажа строится embedding по вручную извлечённым голосовым профилям. Для каждого diarization-спикера строится embedding по его сегментам. Затем считается cosine similarity между спикером и каждым персонажем.

Рабочий validation-порог:

```bash
--similarity-threshold 0.48
```

Если сходство ниже порога, спикер получает метку `unknown_speaker`. При экспорте она отображается как `unknown`.

### 2.7. Экспорт результатов

`src/pair_formatter.py` агрегирует предсказания внутри окна в две колонки:

```text
opening
closing
```

`src.export_detailed_pairs.py` создаёт плоскую таблицу событий `detailed_pairs.xlsx`, удобную для ручного просмотра результатов по тестовому фильму.

### 3.1. Сравнение режимов извлечения intent

Результаты получены для версии **v11** (`--intent-mode combined`, `--ml-confidence-threshold 0.35`, `--similarity-threshold 0.48`, `--diarization-segment-mode regular`) с ML-классификатором **RidgeClassifier + CalibratedClassifierCV**.

| Режим | Pred | Jaccard | Precision | Recall | F1 | matched | exact |
|---|---:|---:|---:|---:|---:|---:|---:|
| Rule-based | 28 | 0.169 | 0.393 | 0.229 | 0.290 | 18 | 6 |
| ML only | 42 | 0.098 | 0.190 | 0.167 | 0.178 | 22 | 6 |
| Combined | 52 | 0.136 | 0.231 | 0.250 | 0.240 | 34 | 8 |

Rule-based режим имеет самый высокий scalar F1 из-за более высокой точности. Combined-режим выбран как рабочий, потому что он даёт больше найденных совпадений (`matched=34`) и больший recall. Для итогового тестового файла это важнее, чем максимальная строгость rule-based режима.

## 4. Тестовый фильм «Питер FM»

Обработка выполняется в режиме `--scan-windows`: pipeline сканирует уже подготовленные чанки, не читая validation Excel.

Итоговые шаги:

1. Извлечь голосовые профили персонажей через `src.extract_audio_profile.py`.
2. Разбить фильм на чанки через `src.chunk_film.py`.
3. Выполнить ASR в Colab.
4. Выполнить diarization в Colab.
5. Запустить локальный постпроцессинг через `src.pipeline.py`.
6. Экспортировать `detailed_pairs.xlsx` через `src.export_detailed_pairs.py`.
7. Оценить результат через `notebooks/evaluation.ipynb` (gold: `data/raw/gold/gold.xlsx`).

Итоговые файлы:

```text
artifacts/test_piter_fm/detailed_pairs.xlsx
artifacts/test_piter_fm/extracted_pairs.xlsx
```

### 4.1. Сравнение версий на тестовом фильме

После появления gold-разметки (`data/raw/gold/gold.xlsx`, 27 уникальных пар) обе версии pipeline были оценены через `notebooks/evaluation.ipynb`.

**Ключевые отличия версий:**

| Параметр | Предыдущая версия | Текущая версия (v11) |

| Количество событий | 403 (361 уникальная пара) | 102 (87 уникальных пар) |

**Результаты оценки:**

### 4.2. Анализ разницы между версиями

Recall и число точных совпадений (exact) одинаковы у обеих версий: обе находят одни и те же 10 пар из 27 gold. Разница — только в числе ложных срабатываний.

Замена LogisticRegression на RidgeClassifier сократила число предсказаний в 4.1 раза (361 → 87) при неизменном recall, что дало рост precision в 4.1 раза (0.028 → 0.115) и рост F1 в 3.4 раза (0.052 → 0.175).

Причина: RidgeClassifier с `CalibratedClassifierCV` имеет лучше откалиброванные вероятности, поэтому при пороге 0.35 он даёт меньше уверенных предсказаний на неоднозначных коротких репликах («давай», «ну», «ладно»), которые LogisticRegression чаще классифицировал как contact_open или contact_close.

Recall остался низким (0.37 на all, 0.50 на opening) по тем же причинам, что и на validation: часть gold-событий — прагматические реплики без однозначных маркеров контакта, а часть теряется из-за ошибок speaker attribution.

## 5. Анализ ограничений текущего решения

### 5.1. Низкая полнота

Даже combined-режим на validation даёт recall 0.250. Это означает, что значительная часть gold-событий не попадает в итоговые предсказания. Причины:

- часть contact-событий выражена не формульными приветствиями/прощаниями, а прагматическими репликами;
- ML-классификатор обучен всего на 1170 репликах;
- validation-фильм отличается от фильмов обучающей выборки;
- часть реплик может быть искажена ASR или потеряна при сборке utterances.

### 5.2. Ошибки speaker attribution

Итоговая оценка требует точного формата `Спикер - фраза`. Поэтому правильная фраза с неправильным именем спикера всё равно считается ошибкой. Это особенно важно для фильмов с большим числом персонажей и похожими голосами.

В текущем коде используется Resemblyzer и один порог cosine similarity. Если уверенность ниже порога, спикер становится `unknown`.

### 5.3. Ложные срабатывания rule-based компонента

Некоторые короткие слова и конструкции могут быть контактными только в контексте:

```text
давай
ну
алло
спасибо
можно
```

Rule-based компонент частично решает это через веса, ограничения на позицию и списки неоднозначных правил, но такие случаи остаются источником ошибок.

### 5.4. Неоднозначность самой разметки

В gold-разметке встречаются не только прямые фразы типа `привет` или `пока`, но и прагматические открытия/закрытия контакта:

```text
Желаете что-нибудь?
Я пошёл
Буду через полчаса
Надумаешь? Звони
```

Поэтому задача шире, чем простое обнаружение приветствий и прощаний. Это объясняет, почему одного лексикона недостаточно, а ML-компонент даёт дополнительное покрытие.

## 7. Направления для улучшения

### 7.1. Speaker attribution

Ошибка в имени спикера делает правильно найденную фразу промахом. Это, вероятно, наибольший источник потерь recall.

- **Снизить порог similarity** (`--similarity-threshold`) с 0.48 до 0.40–0.42: больше реплик получат имя вместо метки `unknown`, ценой небольшого роста ошибочных атрибуций.
- **Расширить голосовые профили**: добавить больше чистых сегментов речи каждого персонажа, включая разные акустические условия (тихие сцены, диалоги по телефону).
- **Заменить Resemblyzer** на более современный энкодер — x-vector (SpeechBrain) или ECAPA-TDNN: они точнее работают на коротких сегментах и при фоновом шуме.

### 7.2. Покрытие прагматических реплик

Часть gold-событий — не формульные приветствия, а прагматические открытия и закрытия контакта, которых нет в лексиконе и которые редко встречаются в обучающей выборке.

- **Расширить `MANUAL_PATTERNS`**: вручную добавить устойчивые конструкции из ошибок (false negatives) тестового и validation-фильмов — быстро и без переобучения.
- **Снизить `--ml-confidence-threshold`** с 0.35 до 0.25–0.30: ML начнёт предсказывать на пограничных случаях, за счёт роста ложных срабатываний.
- **Дообучить на разметке тестового фильма**: добавить gold-события «Питер FM» в обучающий JSONL и переобучить модель — это напрямую закрывает пропуски, специфичные для жанра и стиля фильма.

### 7.3. Объём и разнообразие обучающих данных

Текущая обучающая выборка — 1170 реплик из 18 фильмов. Этого недостаточно для уверенного обобщения на новые фильмы.

- Добавить разметку из новых фильмов разных жанров и эпох, особенно с большим числом примеров `contact_close` (их в выборке втрое меньше, чем `contact_open`).
- Применить аугментацию: перефразирование, опечатки, разные формы обращения — это дёшево и улучшает устойчивость к ASR-ошибкам.

### 7.4. Качество ASR и сборки реплик

Искажённая транскрипция или неверно собранная реплика не может быть корректно классифицирована вне зависимости от модели.

- Проверить в `windows/*/utterances.jsonl`, присутствуют ли нужные реплики из gold с правильным текстом — это позволит разделить потери ASR и потери классификатора.
- Попробовать модель ASR большего размера (`large-v3` вместо `medium`) для коротких и тихих реплик.
- Скорректировать параметры сборки utterances (`--max-pause-within-utterance-sec`, `--min-utterance-duration-sec`) если короткие контактные реплики теряются при объединении сегментов.
